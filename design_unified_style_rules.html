1 · Problem

Theme YAMLs currently spread styling concerns across seven top-level sections that each have their own grammar. text_styles embeds yet another rule kind (size_rules) inside the value of a single style. The result is hard to teach, hard to override, and forces engine code to know about each section's bespoke shape.

Concrete pain points

  • Five token-definition sections (text_styles, box_styles, line_styles, icon_styles, plus element_styles bindings) — each with its own keys.
  • size_rules is a rule list nested inside a token; it cannot reuse the global selector grammar (no event_type, no federal_holiday).
  • style_rules, swimlane_rules, and base.size_rule all match-and-apply, but use three different shapes.
  • No single place a theme author can grep for "what styles this element."
  • box_styles and the day_box rule target are the same concept (a filled, stroked rectangle, optionally with a pattern) but use two different property vocabularies (fill vs fill_color; patterns only on day_box).

What "consolidate" means here

  • One top-level list — style_rules — covers all styling intent.
  • A rule has the same three parts everywhere: select, apply_to, style.
  • Named tokens (heading, cell, grid…) are still definable, but as ordinary rules with define:, not as a separate section.
  • Selectors can stack predicates: papersize and event_type and nonworkday in the same rule.
  • The day box is just a box. No separate day_box target — day cells are styled as box:day (or via the ec-day-box element), with the same property vocabulary as every other box.
  • Durations, vertical-line bands, and swimlane regions are also just boxes. They become box:duration, box:vline, box:swimlane_heading, box:swimlane_content. The only non-box thing left in the swimlane area is the routing rule (which row goes into which lane) — that stays as a one-property assignment.

3 · Design principles

  1. One shape, one place. Every styling decision is a rule with the same three keys: select, apply_to, style.
  2. Definition is just a rule. Defining the heading text token is a rule that selects nothing and applies to a named token slot. Overriding it for a paper size is another rule that adds a predicate.
  3. Selectors compose freely. Any selector key may combine with any other (papersize + event_type + federal_holiday). No more grammar islands.
  4. Order is the only precedence. Later rule wins. No surprise priority axes. Themes can extends: for layering.
  5. Non-styling config stays put. Format strings, fiscal semantics, layout dimensions, and structural declarations are not rules and remain in their existing top-level sections. See §3.1 for the full inventory.

3.1 · What's not a rule — non-styling config

Not every value in a theme YAML is about appearance. Many describe what calendar to compute, how to label it, or how to lay it out — questions a selector and a style bag cannot sensibly answer. Trying to fold them into style_rules would either invent meaningless predicates ("apply this fiscal year offset when…") or pollute the rule schema with one-off keys. They stay in their existing sections (theme, base, events, fiscal, weekly, timeline, blockplan, mini_calendar, …) under the same keys.

Test for whether a value belongs in style_rules: can you finish the sentence "draw this surface with…"? If yes, it's a rule. If instead the value answers what to compute, what to label, what to lay out, what structures exist, or which algorithm to run, it's non-styling config.

Boundary cases — what looks like config but is actually styling

A few keys in legacy themes look like config because of where they sit in the YAML, but they are styling and migrate into style_rules:

  • colors.federal_holiday.color / colors.company_holiday.color — styling. Becomes a style_rules entry on box:day with select: { federal_holiday: true }.
  • weekly.day_box.stroke_color, .fill_color, .number_color, etc. — styling. Captured by the define: box, as: day rule and a define: text, as: day_number rule.
  • timeline.today_line_color / today_label_color / axis_color / tick_color — styling. Move to box:vline / text:today_label / line:axis tokens (or scoped rules).
  • blockplan.timeband_fill_color, timeband_label_color, etc. — styling. Move to per-band style_rules keyed by select.band.

The structural mate of each item — e.g. timeline.today_date: "20260315", blockplan.timebands[i].label, the swimlane name — stays put. Style and structure separate cleanly along the test in the callout above.

Net result: after migration, weekly, timeline, and blockplan shrink to small, config-only sections — format strings, fiscal semantics, geometry, and the lists of bands and swimlanes. Anything that controls appearance lives in one place: style_rules.

4 · Unified rule schema

style_rules: list<Rule>

Rule:
  name: str # human label, used in error messages
  define: text | box | line | icon # optional: declare a named token
  as: str # required when `define` set; the token name
  select: Selector # optional: empty match = always
  apply_to: Target | list<Target> # required (unless `define` is set). List form fans out — see §4.5
  style: map<str, any> # the property bag

Selector:
  # context
  papersize: str | list<str>
  visualizer: weekly | timeline | blockplan | mini | …
  scope: base | header | footer | watermark | …
  # element identity
  element: str | list<str> # ec-heading, ec-day-box, …
  # content predicates (existing style_rules vocabulary)
  event_type, task_name, notes, resource_group,
  resource_names, wbs, milestone, rollup,
  federal_holiday, company_holiday, nonworkday,
  priority | priority_min | priority_max,
  percent_complete: {min, max}, band, value, repeat

Target (apply_to): one of
  # token slots (replaces text_styles / box_styles / …)
  text:<name> · box:<name> · line:<name> · icon:<name>
  # element classes (replaces element_styles)
  element
  # non-styling assignments (still rules, but no `style:` properties drawn)
  lane # routing: style: { swimlane: <name> }
  # everything visual is a box. Stock token names:
  # box:day · box:duration · box:vline
  # box:swimlane_heading · box:swimlane_content
  # box:cell · box:header · box:callout · box:highlight · …
The single mental model: a rule is a select → apply_to → style triple. define: is a shorthand that creates a token slot; everything else references that slot or applies styling directly to a target.

4.1 · Why day_box folds into box

A day cell is a rectangle with a fill, a stroke, an optional pattern overlay, and a content payload — the same primitive as every other box token. Yet the legacy schema models it as a fourth target type with its own property names (fill_color / pattern_*) that diverge from box_styles (fill, no patterns). That divergence is the source of three real problems:

  • Patterns are box-shaped, but only day boxes can have them. A theme author who wants a striped header cell or a hatched callout has no path; the property simply isn't recognized on box_styles.
  • Two name spellings for the same concept. fill in token definitions, fill_color in day-box rules. Theme authors hit this every time they copy a value from one section to another.
  • Two code paths in the renderer. Day-box drawing has its own decoration pipeline; other boxes go through the element-style path. Bug fixes have to land in both.

The unification

  1. Pattern properties become box properties. Every box:<n> accepts pattern, pattern_color, pattern_opacity. The day box loses no expressive power; every other box gains pattern support.
  2. The day cell is a named token: box:day. Default themes ship with define: box, as: day setting the baseline fill/stroke that weekly.day_box.* used to control. The ec-day-box element class binds to it.
  3. Existing day-box content rules retarget to box:day. A rule like "federal holidays get a tomato pattern" now reads apply_to: box:day with a content selector — same predicate vocabulary, no special target.
  4. Property names harmonize on fill/stroke. The loader accepts the legacy fill_color and stroke_color spellings as aliases for one release, then deprecates them.
Net effect: the target list drops from four target families (token, element, day_box, content surfaces) to three (token, element, content surface). Patterns become a first-class box feature usable on cells, callouts, headings — anywhere a box is drawn.

4.2 · Why duration, vertical_line, and swimlane regions also fold into box

The same argument that retired day_box applies to the remaining content surfaces. Each one is, visually, a filled-and-stroked rectangle that happens to be drawn in a different layout slot. Their schemas diverged only because each was added independently.

Old targetWhat it actually drawsNew target
duration A horizontal bar with a fill, stroke, optional dash, and a date label hanging off the end box:duration (+ text:duration_date for the label)
vertical_line A vertical band with a stroke on one edge and a (possibly alternating) fill box:vline — the "line" is just a stroke on a thin box
swimlane (visual) A heading cell + a content area, separated by a divider, with a label inside box:swimlane_heading + box:swimlane_content
swimlane (routing) "This row belongs to lane Engineering" — no visual output apply_to: lane with style: { swimlane: Engineering }

Property mapping

The legacy keys map onto the unified box vocabulary cleanly. The loader accepts both spellings during the deprecation window:

Old keyNew keyNotes
fill_colorfillscalar or list (lists drive alternation, see below)
stroke_colorstroke
stroke_dasharraydasharraysame SVG syntax
timeline_fill_colorfill on box:swimlane_contentwas a separate field only because it lived inside the swimlane definition
label_color, label_align_h/von text:swimlane_labellabel styling is a text token, not a box property

Placement vs. styling. A handful of keys (align on vlines, split_ratio on swimlanes) describe where the box is drawn, not what it looks like. They live alongside style properties in the same rule's style: bag for now — a future cleanup can promote them to a placement: sub-key if the distinction earns its keep.

What this buys

  • One renderer primitive. _draw_box(target, ctx) is the only path for any rectangular fill+stroke+pattern surface. Day cells, duration bars, vline bands, swimlane headings — all collapse into one code path.
  • Patterns everywhere. Hatched duration bars, striped swimlane headings, dotted vline bands — all become trivial since pattern properties live on the box vocabulary.
  • Symmetric rule shapes. Every visual content rule reads apply_to: box:<name> + select: … + style: {fill, stroke, pattern, …}. There is no second grammar to learn.
  • Routing is the only non-box concern in the swimlane area, and it surfaces as a single apply_to: lane target with one property (swimlane:) — clearly separated from styling.

4.4 · Milestones & single-day events

The day box, the duration bar, the vline band, and the swimlane region cover the container surfaces. Two content types still need a styling story: single-day events (the row inside a day cell on weekly, or the callout on a timeline) and milestones (the diamond/flag marker plus its label). Today these are reachable only through generic element bindings (ec-event-name, ec-milestone-marker, etc.) — there is no way to say "this milestone gets a red flag and bold label" without forking the renderer.

The new tokens

TokenWhat it stylesProperty family
box:event Single-day event row inside a day cell · timeline event callout full box vocab (fill, stroke, pattern, dasharray, fill_opacity, …)
box:milestone Optional halo / background box behind a milestone marker full box vocab
icon:event (existing) The leading icon on an event row icon, color, size
icon:milestone The milestone marker glyph itself (diamond, flag, star, …) icon, color, size, rotation
text:event_name (existing token) Event name label text vocab
text:event_notes (existing token) Event secondary text text vocab
text:milestone_label Text rendered next to a milestone marker text vocab
Why a box:milestone at all? The marker glyph is an icon, but themes often want a tinted halo or a colored callout behind it (e.g. red background for a critical milestone). Modeling that as a separate box:milestone keeps the marker glyph and its background independently overridable, with no special grammar.

Selectors that apply to events & milestones

The same selector grammar from existing day-box rules carries over — no new keys. The renderer simply matches against the data attached to the event/milestone being drawn:

SelectorNotes
event_type: event | milestone | durationDiscriminates the three content kinds.
milestone: true | falseShorthand for event_type: milestone.
task_name, notesSubstring match against the event's name / notes.
priority · priority_min · priority_maxNumeric. Supports the same int-or-list / range form as blockplan swimlanes.
resource_group, resource_names, wbsRouting-style metadata on the event.
percent_complete: {min, max}Range match (e.g. {min: 100} → completed).
rollup: trueMatch rollup/parent tasks vs. leaves.

What this unlocks: theme authors can decorate any subset of milestones or single-day events — by name, priority, completion state, resource group, or visualizer context — using the same selector grammar already used for day boxes. No new keys, no new sections.

Day-box rules vs. event/milestone rules — keeping them straight

Both can use the predicate milestone: true. The difference is the target:

  • apply_to: box:day, select: { milestone: true } — styles the host day cell on a day that contains a milestone (e.g. tint the day gold).
  • apply_to: icon:milestone, select: { … } — styles the milestone marker itself.
  • apply_to: text:milestone_label — styles the label text next to the marker.
  • apply_to: box:milestone — styles the optional halo behind the marker.

The selector grammar is identical; the target picks the surface. This is the same pattern used for durations: box:day tints the host cell, box:duration styles the bar.

4.5 · Applying one rule to multiple targets

Several earlier examples (completed events, day-box + cell binding, etc.) used a list-valued apply_to:. This subsection formalizes that mechanism — what it means, when to use it, and what it doesn't do.

Syntax

apply_to: accepts either a single target string or a list of target strings. Both forms are first-class — the list form is sugar for N copies of the same rule, one per target:

list form (sugar) 
- name: muted completed items
  apply_to: [box:event, box:duration, text:event_name]
  select: { percent_complete: { min: 100 } }
  style:
    fill: "#f4f4f4"
    color: grey
    italic: true
expanded equivalent 
- name: muted completed items (1/3)
  apply_to: box:event
  select: { percent_complete: { min: 100 } }
  style: { fill: "#f4f4f4", color: grey, italic: true }
- name: muted completed items (2/3)
  apply_to: box:duration
  select: { percent_complete: { min: 100 } }
  style: { fill: "#f4f4f4", color: grey, italic: true }
- name: muted completed items (3/3)
  apply_to: text:event_name
  select: { percent_complete: { min: 100 } }
  style: { fill: "#f4f4f4", color: grey, italic: true }

Property routing across targets

Not every key in style: applies to every target. fill is meaningful for a box but not for a text; italic is meaningful for text but not for box or icon. The rule is:

  • Each target consumes only the keys it recognizes. Unknown keys are silently dropped per target, not for the whole rule.
  • Load-time validation. Every key in style: must be recognized by at least one target in the list. A key recognized by none is a typo — the loader emits a warning and (in strict mode) fails.
  • No silent shadowing. If two listed targets both recognize a key (e.g. both box:event and box:duration consume fill), they each apply it independently. Fan-out is per-target, not first-wins.
Why this works: the property vocabularies for box, text, icon, and line are nearly disjoint by design. The handful of overlaps (color on text and icon; size on text and icon) carry the same intent across both. So the routing rule is "send each key to every target that knows it" — no per-target sub-bag is needed for the common case.

Precedence: a fanned-out rule counts as N applications

The Pass-3 resolver from §6 walks style_rules in order and applies each match to the rendered surface. A list-valued rule contributes one application per (target, position) pair. Concretely:

  • Two rules — rule A with apply_to: [box:event, text:event_name] followed by rule B with apply_to: text:event_name — produce three applications: A→box:event, A→text:event_name, B→text:event_name. B wins on text:event_name (later); A is the only rule on box:event.
  • Within a single rule, the order of the target list does not matter — the targets do not collide with each other (they are different surfaces).

Multi-target vs. multi-selector — keep them straight

Two superficially similar shapes solve different problems:

GoalMechanismExample
One style, several surfaces List-valued apply_to apply_to: [box:event, text:event_name]
One style, several element classes (CSS-like) Single target element + list-valued select.element apply_to: element, select: { element: [ec-day-box, ec-cell] }
One style, several matching values on one selector Single target + list-valued selector predicate select: { task_name: [Release, Cutover] }

The first writes to multiple kinds of surface in one breath. The second and third match multiple data items against a single surface. They compose freely: a list-valued apply_to alongside a list-valued selector simply fans out the cross-product.

Restrictions

  • No list with define:. A definition rule registers exactly one token, so apply_to is implicit and a list is meaningless. The loader rejects define: + list apply_to:.
  • No wildcards (yet). box:* or text:* is not accepted. Listing the targets explicitly keeps the impact of a rule grep-able.
  • Mixing lane with visual targets is allowed but unusual. A list of [lane, box:swimlane_heading] would assign routing and tint the heading in one rule; readable for some authors, opaque for others. Allowed; not recommended.

What you cannot do — and the future-work alternative

There is no per-target style sub-bag in v1. If you need to send different property values to different targets, write separate rules. A heavier alternative — keyed style bags like style: { box:event: { fill: red }, text:event_name: { color: red } } — is deferred; the property-routing rule covers all observed real-world cases in the existing themes, and the keyed form would muddy the otherwise-flat style: shape.

Rule of thumb: reach for list-valued apply_to when the same visual treatment (a tint, a fade, a state indicator) needs to land on more than one surface simultaneously. Reach for separate rules when each surface needs a different property value, even if the predicate is shared.

6 · Resolution & precedence

The theme engine builds the resolved style for any (element, context, content) tuple in three passes:

  1. Pass 1 — token registry. Walk style_rules in order. For every rule with define:, register the token under <kind>:<as> with its style bag. Definitions with a select are stored as conditional layers; later definitions of the same name simply append.
  2. Pass 2 — token resolution. For each requested token (e.g. text:heading at papersize=letter), merge the matching layers in order. This collapses the old size_rules behavior with no special case.
  3. Pass 3 — application. For the current draw call, walk all rules whose apply_to matches the surface (element, day_box, swimlane, …) and whose select matches the context. Rules with style.use: pull from the resolved token; rules with raw properties override them. Last match wins.
Determinism: a single ordered list means no priority field, no ambiguity from cross-section interaction. To override a base theme value, append a rule. extends: simply prepends the parent theme's rules to the child's.

6.1 · An empty select matches always

This makes "definition" a degenerate case of "application" — no special grammar.

6.2 · What if I just want to read the heading style?

The engine exposes theme.resolve(token, context). Renderer code calls theme.resolve("text:heading", ctx) and gets a fully merged dict — same API it uses today against text_styles.

7 · Migration — single clean break

There is no phased rollout, no parallel legacy parser, and no deprecation window. After this design ships, the only theme schema the engine recognizes is the unified one. All in-tree themes are converted as part of the implementation PR; any external themes are converted by running the one-shot converter once.

Why a clean break. A dual-path loader doubles the surface area we have to test, and the legacy schema is the very thing this design exists to retire. Carrying it forward as "supported but discouraged" preserves all the cost (two grammars, two code paths, two sets of tests) and keeps the consolidation from finishing. We take the cost once, in one PR.

7.1 · The implementation PR — what lands at once

  1. New loader. Parses unified style_rules only. Legacy section names (text_styles, box_styles, line_styles, icon_styles, element_styles, swimlane_rules, embedded size_rules, base.size_rule) become hard parse errors with a message that points at the converter.
  2. New target/property vocabulary only. No property aliases — fill_color, stroke_color, stroke_dasharray, timeline_fill_color are not recognized anywhere in the codebase. Same for the retired targets (day_box, duration, vertical_line, swimlane): unrecognized.
  3. Renderer collapse. One _draw_box(target, ctx) primitive handles fill, stroke, dash, pattern, and (via list-valued fill) alternation. The day-cell, duration-bar, vline-band, and swimlane heading/content draw sites all funnel through it. The legacy day-box decoration pipeline, the duration-specific draw path, and the milestone-marker line-style fallback are deleted.

  5. Docs and SAMPLE.yaml rewritten. Every example switches to the unified form. No "legacy reference" appendix.
  6. One-shot converter. tools/migrate_theme.py reads a legacy YAML and emits the unified form. Used to produce the conversions in step 4 and shipped for external consumers to run once. Not wired into the loader and not depended on at runtime.

Inline shorthand (apply_to: text:heading@small) no Adding a second grammar to express what select already expresses. 9.12 · Reconciling the per-element text: sub-bag

USER_GUIDE.md lines 1117–1173 document a feature the unified design needs to retire deliberately — not silently. The current style_rules accepts a nested text: block inside a content rule's style: bag, keyed by the role of the text within the matched target: 

style_rules:
  - name: Priority 1
    apply_to: [event, duration]
    select: { priority: 1 }
    style:
      stroke_color: crimson
      text:
        event_name:     { font: OfficinaSans-Bold, font_color: crimson }
        duration_name:  { font: OfficinaSans-Bold, font_color: white }

This nested form was one rule with three internal property bags (one for the box, one for event_name text, one for duration_name text). §4.5 of this design explicitly defers per-target style sub-bags — so the nested shape doesn't fit the unified grammar.

Decision

The nested text: sub-bag is removed. It rewrites cleanly into multiple peer rules with the same selector — exactly the "rule of thumb" called out in §4.5: when each surface needs different property values, write separate rules even if the predicate is shared.

Before / after

before (one rule, nested) 
- name: Priority 1
  apply_to: [event, duration]
  select: { priority: 1 }
  style:
    stroke: crimson
    text:
      event_name:
        font: OfficinaSans-Bold
        color: crimson
      duration_name:
        font: OfficinaSans-Bold
        color: white
after (peer rules, flat) 
- name: Priority 1 — boxes
  apply_to: [box:event, box:duration]
  select: { priority: 1 }
  style: { stroke: crimson }

- name: Priority 1 — event name
  apply_to: text:event_name
  select: { priority: 1 }
  style: { font: OfficinaSans-Bold, color: crimson }

- name: Priority 1 — duration name
  apply_to: text:duration_name
  select: { priority: 1 }
  style: { font: OfficinaSans-Bold, color: white }

Converter responsibility

tools/migrate_theme.py walks every style_rules entry that contains a nested style.text block and emits one peer rule per inner key. The flat box/stroke/fill keys at the rule's top level become the original rule's style: (with the box: target list); each inner text-role key becomes a new rule targeting text:<role> with the same selector. The mapping from text-role to target uses the existing role-to-element table at USER_GUIDE.md lines 1158–1172 (the "Text Element Reference" table — event_name → text:event_name, day_number → text:day_number, etc.).

11 · No hardcoded defaults; reference themes; section comments

Once the theme schema is consolidated, two operational rules become workable for the first time: themes must be complete, and themes should be self-documenting. This section spells out the implications: dropping the fallback defaults baked into CalendarConfig, defining the missing-key error format, designating two reference themes (one comprehensive, one minimal) as the source of truth for what a complete theme looks like, and adding section-purpose comments to every migrated theme.

11.1 · Removing hardcoded defaults from config.py

CalendarConfig currently carries roughly 100+ fields with non-None defaults — colors, font names, numeric geometry, format strings. The styling fields go away with the style_rules migration (§7.1). Of what remains — non-styling config — every default is also removed. The dataclass declares the field's type and required visualizers; it never declares a fallback value.

Why drop them

  • Hidden defaults make themes ambiguous. A theme that doesn't set mini_calendar.title_format renders as "MMMM YYYY" today because the dataclass says so. A reader of the theme file has no way to know what they'll get without consulting the source. Removing the default makes the theme either explicit or invalid.
  • The reference themes become the source of truth. When the dataclass is silent, SAMPLE.yaml (complete) and basic.yaml (minimal) are the only places to look up a sensible value. That's a documentation boundary that doesn't drift.
  • Validation gets sharper. The loader can statically prove a theme is complete for a given visualizer before any rendering work begins — no surprise mid-render fallbacks.
  • Theme inheritance is honest. A theme that extends: the default theme inherits exactly what's written in the parent, not a phantom dataclass layer underneath.

What changes in CalendarConfig

  • Every field's field(default=…) / field(default_factory=…) is replaced with no default. In a frozen dataclass: the field is required at construction.
  • Each field is annotated with the visualizers that consume it (a small set / enum). The loader uses this to scope the missing-key check by visualizer (see §11.2).
  • Initialization stops being "instantiate CalendarConfig() with defaults, then overlay theme values." Instead, the loader builds the config from the theme YAML directly, raising on missing required keys before any field is read.

11.2 · Error reporting for missing required settings

A missing required key produces a single, actionable error that names the key, gives an example value, says which visualizer needs it, and points at the reference theme that demonstrates it. The format is fixed: 

error: theme is missing a required setting for visualizer 'mini'

  missing key:   mini_calendar.title_format
  type:          str
  example value: "MMMM YYYY"
  used by:       mini, mini-icon
  reference:     config/themes/SAMPLE.yaml line 749
                 config/themes/basic.yaml  line 41

  add to your theme:

    mini_calendar:
      title_format: "MMMM YYYY"

Behaviors

  • Lazy by visualizer. A theme used only with weekly never has to satisfy compact_plan.* keys. The loader knows the active subcommand and only checks fields whose used_by set intersects it. The catalog of keys per visualizer comes from the dataclass annotations.
  • All missing keys at once. The loader does not stop at the first error — it collects every missing key for the active visualizer and reports them together, so an author fixing a theme makes a complete pass instead of a play-debug-rerun loop.
  • Example values come from basic.yaml. The example shown in the error is the literal value at the same key in basic.yaml — keeping the error message and the reference theme in sync without a parallel registry.
  • Style-rule keys are checked separately. If a theme is missing a token the renderer expects (e.g. no define: text, as: day_number when a weekly render is asked for), the same error format applies, with the missing target written as the key (style_rules: missing definition of text:day_number).
  • Pure schema errors are distinguishable from value errors. A missing key, an unknown key, and a type mismatch all use the same error template but with a different leading word: missing, unknown, invalid. This is what the §7.1 loader-rejection behavior promises in concrete form.

11.3 · SAMPLE.yaml — complete reference theme

The current SAMPLE.yaml is a partial reference; some sections are commented out or omitted, and the theme cannot drive every subcommand by itself. After migration:

  • Every required key for every visualizer is present and uncommented. A user can run ecalendar.py weekly … --theme SAMPLE, … mini, … mini-icon, … text-mini, … timeline, … blockplan, … compactplan, and … excelheader against the same SAMPLE.yaml and get a rendered output for each — no missing-key errors.
  • Every style_rules token used by any renderer is defined. The defaults are conservative: neutral colors, sensible fonts, no decorative patterns. The intent is "demonstrably complete," not "stylistically opinionated."
  • Optional features have a commented example, not silence. Patterns, hash rules, swimlane fills, milestone halos — present as commented examples that show the shape and key set, so a reader doesn't have to discover them from this design document.
  • Every section gets a purpose comment. See §11.5.
  • Validation in CI: a test invokes every subcommand against SAMPLE.yaml over a small fixture date range; any missing-key error fails the build. This guarantees the "complete" claim doesn't drift as new keys are added.

11.4 · basic.yaml — minimum viable theme

A new file added in the same migration PR. basic.yaml contains the absolute minimum set of values — and only those values — required for every visualizer to produce a non-erroring output. It complements SAMPLE.yaml at the opposite end of the completeness spectrum.

Construction rules

  • One value per required key, period. No commented-out optional examples, no duplicate alternatives, no decoration. If a key isn't required by any visualizer, it isn't present.
  • Values are deliberately plain. Black text, white fills, single-color palettes, standard system fonts (Roboto-Regular family). The output looks intentionally austere — that's the point. A theme author copying basic.yaml as a starting point sees clearly which lines they need to change to develop a visual identity.
  • Section-purpose comments only. Per §11.5 — no other commentary. The file is short enough that the YAML is its own documentation.
  • Doubles as the example-value source for §11.2 errors. When the loader emits a missing-key error, the example value it prints is read from basic.yaml. Editing basic.yaml updates the help text — no parallel string table.

Approximate size

Estimated 80–120 lines: roughly the count of required non-styling keys across all visualizers, plus ~15 style_rules entries (the minimum token definitions and element bindings), plus the section-purpose comment headers. For comparison, SAMPLE.yaml after migration sits in the 600–900-line range — it carries the same baseline plus annotated examples for every optional feature.

Use cases

  1. New theme starter. cp config/themes/basic.yaml config/themes/mytheme.yaml — a known-complete starting point.
  2. Bug-report repro. The shortest theme that exhibits the problem; reduces noise in issue triage.
  3. Schema-completeness test. CI invokes every subcommand against basic.yaml; if anything fails, either the theme is missing a key (fix the theme) or a default crept back into the dataclass (fix the code).
  4. Loader error help. The example value in §11.2's error message is read from this file.
